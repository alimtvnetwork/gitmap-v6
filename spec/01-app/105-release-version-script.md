# Release-Version Install Script (spec 105)

> **Status:** planned for v3.39.0
> **Scope:** dedicated, version-pinned installer used **only** by the release pages.
> **Replaces:** the practice of pointing release pages at the generic `install.ps1`.

## Why this exists

The current `install.ps1` / `install.sh` always resolves to the **latest**
release. That's correct for the front-page "Get Started" button, but it is
wrong for a release page like `/release/v3.36.0`, where the entire point of
the URL is "give me **this** version, not whatever shipped last week."

`release-version.ps1` / `release-version.sh` is a second, narrower script
whose contract is: **install exactly the version it was asked for, never
auto-upgrade, never silently fall back.**

## URL → version flow

```
User opens https://gitmap.dev/release/v3.36.0
        │
        ├─ Release page parses :version from the URL
        │
        ├─ Renders a one-liner pre-filled with that version:
        │     iwr https://gitmap.dev/scripts/release-version.ps1 | iex
        │     # then internally: Install-GitmapVersion -Version v3.36.0
        │
        └─ User pastes & runs → script downloads exactly v3.36.0
```

## Distribution strategy: BOTH generic + snapshot

Two artefacts ship for every release:

| Artefact | Where it lives | Purpose |
|----------|---------------|---------|
| `release-version.ps1` (generic) | `/scripts/release-version.ps1` on the docs site | Always-current installer; reads `-Version` parameter |
| `release-version.ps1` (snapshot) | Uploaded as a release asset on every GitHub release | Per-version frozen copy with `$Version = 'vX.Y.Z'` baked at line 1 |

Same for `.sh`. Release pages link to the **snapshot** by default (zero-arg
one-liner that's drift-proof a year from now); the generic script is the
fallback for power users who script their own installs.

### Generic form

```powershell
param(
    [Parameter(Mandatory=$true)] [string]$Version,
    [switch]$AllowFallback,
    [switch]$Quiet
)
```

### Snapshot form

```powershell
# release-version.ps1 — frozen for v3.36.0
$Version = 'v3.36.0'
# ... rest of the body is identical to the generic ...
```

The snapshot is generated by the release pipeline: copy the generic, prepend
the `$Version` line, upload as a release asset.

## What the script does

1. **Validate** `$Version` matches `^v\d+\.\d+\.\d+(-[a-z0-9.]+)?$`. Reject otherwise.
2. **Detect OS/arch** (`windows-amd64`, `linux-amd64`, `darwin-arm64`, etc.).
3. **Resolve the GitHub release**:
   `GET https://api.github.com/repos/<owner>/gitmap/releases/tags/<version>`.
   - On 404 → trigger the [Missing-version flow](#missing-version-flow).
   - On 200 → pick the asset matching the detected OS/arch.
4. **Download** the asset to a temp directory with a SHA256 verification step
   (checksums file is also a release asset).
5. **Extract** the zip into the install directory
   (`$env:LOCALAPPDATA\gitmap\bin` on Windows, `~/.local/bin` on Unix).
6. **Add to PATH** if not already present. Persist to user shell profile.
7. **Chain `gitmap self-install`** — the freshly-installed binary takes over
   to set up completion, profiles, etc. The script's job ends here.
8. **Print verification**: run `gitmap --version` and confirm it matches the
   requested `$Version`. Exit non-zero on mismatch.

## Missing-version flow

If the GitHub release for `$Version` doesn't exist:

1. Print a clear error stating which version was requested and that it's not
   available.
2. **Interactive prompt**: list the 5 most recent valid releases (newest
   first) and ask the user to pick one or quit. Default is **N (quit)**.
3. If running non-interactively (no TTY, or `-Quiet` set, or stdin is closed)
   → skip the prompt entirely and exit 1.
4. If `-AllowFallback` is explicitly set, use the newest patch in the same
   `vMAJOR.MINOR` series without prompting (e.g. `v3.36.0` requested but
   missing → use `v3.36.5` if it exists; else hard-fail).

## Flags

| Flag | Effect |
|------|--------|
| `-Version <vX.Y.Z>` | **Required for generic.** Ignored in snapshot (already baked). |
| `-AllowFallback` | Permit same-minor-series patch fallback (no interactive prompt). |
| `-Quiet` | Suppress prompts and progress output; non-interactive failure mode. |
| `-InstallDir <path>` | Override default install directory. |
| `-NoPath` | Skip PATH modification. |
| `-NoSelfInstall` | Skip the chained `gitmap self-install` step (download + extract only). |

## Exit codes

| 0 | Installed and verified |
| 1 | Requested version not found and user declined fallback |
| 2 | Network / download error |
| 3 | Checksum mismatch |
| 4 | OS/arch unsupported |
| 5 | PATH update failed (binary still installed; warning, not fatal) |
| 6 | `self-install` chained step failed |
| 7 | Version verification mismatch (downloaded binary reports a different version) |

## What this script does NOT do

- **Never** queries `/releases/latest`. The word "latest" does not appear in
  the source.
- **Never** writes to `gitmap-update-channel` or any auto-update config.
- **Never** chains `gitmap self-update`.
- **Never** silently substitutes a different version. Any substitution is
  explicit (interactive choice or `-AllowFallback` flag).

## Release-page integration

Each `/release/:version` page renders two install boxes:

```
╭─ Pinned install (recommended) ────────────────────────────╮
│  Windows:                                                 │
│    iwr https://github.com/.../v3.36.0/release-version.ps1 │
│      | iex                                                │
│                                                           │
│  macOS / Linux:                                           │
│    curl -sSf https://github.com/.../v3.36.0/release-      │
│      version.sh | bash                                    │
╰───────────────────────────────────────────────────────────╯

╭─ Generic install (advanced) ──────────────────────────────╮
│  iwr https://gitmap.dev/scripts/release-version.ps1 | iex │
│  # then: Install-GitmapVersion -Version v3.36.0           │
╰───────────────────────────────────────────────────────────╯
```

The pinned box uses the snapshot asset; the generic box uses the always-
current script with an explicit `-Version` argument.

The front-page Get Started box continues to use the existing `install.ps1`
(latest-resolving) and is **not** affected by this spec.

## Open questions (answered)

- **Q:** Should snapshot or generic be the default link on release pages?
  **A:** Snapshot. Drift-proof for old release pages indexed by search engines.
- **Q:** Should missing-version fall back to latest?
  **A:** No. Interactive prompt with explicit choice, or hard-fail in CI.
- **Q:** Should the script chain `self-install`?
  **A:** Yes (matches first-run experience), but provide `-NoSelfInstall` escape hatch.
